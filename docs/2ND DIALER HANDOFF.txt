UPDATED PROJECT HANDOFF
DOMINION HOME DEALS / SENTINEL / DIALER
Trajectory reset based on the updated source documents and the implementation work completed through the current session
North star: build a two-person operating machine that closes more deals with less support labor. The dialer exists to improve seller conversations, follow-up reliability, context recall, and post-call discipline — not to become software theater.
1. Executive orientation
This handoff supersedes the earlier dialer-only handoff where they conflict. It keeps the original boundary rules, but updates the trajectory using the newer anticipatory strategy documents and the implementation work already completed. The goal is not to preserve old PR sequencing for its own sake; the goal is to keep the dialer on the shortest path to more seller conversations, cleaner follow-up, stronger memory, and more contracts per Logan-hour.
2. Business objective and decision filter
Everything in this dialer effort should still be filtered through one business question: does it improve contracts per founder-hour, speed-to-lead, follow-up reliability, trust, or monetization enough to justify the complexity?
Do not optimize for anti-AI ideology.
Do not optimize for “AI everywhere.”
Optimize for a small team performing like a much larger operation.
Prefer narrow, high-adoption workflow improvements over broad architecture gestures.
3. Updated product doctrine for the dialer
The dialer should be treated as a conversation operating system. That means it owns the live workspace around seller conversations before curated outputs are promoted into Sentinel.
4. Boundary rules that remain non-negotiable
Client file / CRM = durable truth. Dialer = volatile call-session workspace.
crm-bridge.ts remains the only dialer read path into CRM-owned tables.
publish-manager.ts remains the only dialer write path back into CRM-owned tables.
master-client-file-modal.tsx must not absorb live call UI, AI suggestions, live notes, session state, or disposition UI.
Twilio status/session-sync routes remain internal mirror/sync routes, not secondary business-logic centers.
Be bold in internal intelligence; be conservative in external behavior.

6. What is still missing or only partially real
The liveNotes panel is mostly dead infrastructure today: the UI and realtime subscription exist, but there is still no live transcript/feed writer populating it during calls.
There is no true conversation-native STT path yet. The current AI help is note-based and post-call/late-call, not a live copilot.
Seller memory exists in a limited sense through AI summaries feeding the pre-call brief, but there is no formal seller-memory panel, write-gate policy, or unresolved-objection surface yet.
Lead Detail calling and manual/test dial remain secondary paths and do not offer the same session-backed structured outcome loop as the main Call Lead workflow.
AI control-plane basics are not yet clearly in place for the dialer: traces, prompt versions, eval datasets, review gates, and rollback discipline still need to be made explicit.
The review console idea exists in the strategy, but the dialer still lacks a clear operator/admin review surface for bad summaries, bad extractions, or suggestion quality feedback.
7. Phase assessment right now
Phase 1 should be considered “passed enough to move forward” when these conditions are true in real usage:
Every serious live call on the main path creates a structured, reviewable record.
Curated facts can be published in under 30 seconds post-call.
The next follow-up can be scheduled before closing the tab.
The pre-call brief actually improves the next conversation.
AI-assisted qualification prefill helps more than it harms.
No obvious workflow leak still lets a real lead disappear.
8. Correct trajectory from here
The updated source documents change the sequencing pressure. The dialer should not now drift into random UI refinements or prematurely jump to live streaming just because it sounds more advanced. The right trajectory is:
Lock the current session-backed Call Lead path as the canonical production flow.
Begin Phase 2 with the smallest intelligence-layer addition that materially improves live calling, seller memory, and post-call usefulness.
Add control-plane basics and review gates early enough that AI writeback is traceable and debuggable.
Keep inbound / missed-call / warm-transfer voice edge in sight, but do not let front-office ambitions derail the current dialer workspace build.
Move buyer memory, monetizability, and event feedback earlier in the wider system plan than a normal wholesaler would, because acquisition tooling will commoditize faster than buyer liquidity or trust.

10. Best next Phase 2 interpretation for the dialer
The most credible Phase 2 framing is not “make the dialer an AI agent.” It is “make the dialer feel like a real live call workspace with better memory, better note quality, and better traceability.” The practical implication is:
Use the current operator-note seam as the bridge, because it already exists and is driving extraction and summary quality.
Prefer a practical transcript path before a heavier live-stream path if it gets richer summaries and memory into production faster.
Treat live STT / live notes feed as valuable, but not the first mandatory Phase 2 move if a simpler post-call transcript path delivers more value sooner.
Bring in AI observability early enough that summaries, extraction, routing, and future voice flows can be reviewed, compared, and rolled back.
11. AI control-plane rules now required by the updated strategy
Every meaningful AI workflow should have a run ID, trace log, prompt version, and review/rollback story.
No durable CRM writeback should come from an untraced workflow.
Eval loops should begin first where AI already touches production value: post-call summaries, extraction/prefill, and routing/suggestion quality.
Review tooling is not admin fluff. It is the discipline layer that keeps the intelligence stack from becoming magic or sludge.
12. Anti-goals and traps to avoid
Do not let the dialer become a feature dumping ground because “it is where calls happen.”
Do not respond to every gap by adding a new path. Strengthen the canonical session-backed path first.
Do not build live streaming STT first if a simpler transcript path yields better operator value sooner.
Do not force broad upgrades on manual/test or side paths before the main path is truly excellent.
Do not let AI outputs silently become durable truth without traceability, review, and a narrow publish contract.
Do not over-automate empathy, outbound consumer contact, or trust-sensitive moments.

14. Standard adversarial check to keep using
When asking Claude to inspect or extend the dialer, keep a short adversarial section in the prompt. The purpose is not to waste tokens; it is to surface the most important hidden risk before it reaches production.
15. Final handoff summary
The dialer is now past the point where the right move is more Phase 1 thrashing. The updated source documents push in the same direction as the implementation work: keep Sentinel as the deterministic action core, keep the dialer as the conversation operating system around Logan, and begin Phase 2 as a narrow intelligence-layer build with real traces, review gates, and memory value. That is the correct trajectory to the end goal: a two-person acquisitions machine with better trust, better timing, stronger memory, and more contracts per founder-hour.
Appendix A - Snapshot of the current call-path reality
Canonical production path: session-backed Call Lead on the power dialer.
Secondary paths: manual/test dial and lead-detail cell bridge. These remain deliberately weaker and should not redefine the architecture.
Current strongest operator loop: pre-call brief -> in-call notes -> post-call publish -> callback scheduling -> qualification confirm -> next lead.
Most important unfulfilled promise: live transcript/live notes feed and a more explicit seller-memory/review layer.
Appendix B - One-sentence scoring rule for every next build
If a proposed dialer feature does not clearly do at least one of these five things, delay or cut it: make Logan better live on calls, remove support/admin/research labor, prevent lead leakage, improve monetization of the right leads, or build a compounding local memory moat.
